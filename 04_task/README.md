3 Почему делегирование возможно только в списке инициализации?

1 делегирующий конструктор сначала полностью выполняет тело делегата, и только потом выполняет своё собственное
TrackedString x
TrackedString() делегирует в TrackedString("")
TrackedString(const char* "")  делегирует в  TrackedString(std::string(""))
TrackedString(const std::string& s)  выполняется 1 id=1
char* ctor delegated  выполняется 2 (тело именноо char* ctor)
default ctor delegated выполняется 3 (тело default ctor)
все три записи в логе имеют одинаковый id=1 и одинаковый адрес this — это один и тот же объект

TrackedString y("test")  (char* ctor)
TrackedString(const char* "test")  делегирует в  TrackedString(std::string("test"))
TrackedString(const std::string& s)  выполняется 1, id=2
char* ctor delegated  — выполняется 2

делегат выполняется полностью затем выполняется тело делегирующего конструктора а порядок в логе это записи от самого глубокого делегата к самому верхнему

2 раньше до c++11 логику инициализации выносили во вспомогательный метод 
отличия
вызывается после создания объекта когда все поля уже инициализированы 
делегирующий конструктор выполняет инициализацию сразу во время создания объекта

делегирование это часть процесса создания объекта
вспомогательный метод обычная функция которую теоретически можно вызвать в любой момент что делает ее менее безопасной

поля типа const или ссылки можно задать только в списке инициализации конструктора

3 конструктор либо делегирует другому конструктору, либо сам инициализирует поля, но не может делать оба действия одновременно
список инициализации это где происходит создание и начальная инициализация полей объекта а когда программа входит в тело конструктора, все поля уже созданы и инициализированы
если бы делегирование можно было написать в теле кнструктора, возникла бы проблема что  другой конструктор пытался бы создать объект хотя он уже задан
